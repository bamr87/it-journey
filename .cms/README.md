# `.cms/` — IT-Journey AI-augmented CMS data layer

This directory is the **single source of truth** for the AI-augmented content
management system: the machine-readable view of all site content, the canonical
content schema, and the daily improvement loop's reports and worklists.

It is a dot-directory, so **Jekyll ignores it automatically** (it never ships to
the built site) and tools — the daily loop, CI, and the future VS Code
extension — read it at a stable path.

## What lives here

| Path | Committed? | Produced by | Consumed by |
|---|---|---|---|
| `config.yml` | ✅ (source) | hand-edited | the engine |
| `schema/content-schema.json` | ✅ | `cms.py index` | extension, loop, docs |
| `index/summary.json` | ✅ | `cms.py index` | dashboards, trend tracking |
| `index/content-index.json` | ❌ (regenerated, ~500KB) | `cms.py index` | extension, loop |
| `reports/<date>.md` | ✅ | `cms.py analyze` | humans |
| `worklists/<date>.md` | ✅ | `cms.py plan` | the `cms-curator` loop |

## The engine

All of the above is generated by **[`scripts/cms/cms.py`](../scripts/cms/cms.py)**,
a thin orchestration + indexing layer. It does **not** duplicate the repo's
existing fixers/validators — it aggregates a full-coverage *view* and turns it
into a prioritized, lane-classified worklist.

```bash
python3 scripts/cms/cms.py index     # build index + summary + schema
python3 scripts/cms/cms.py analyze   # + daily Markdown analysis report
python3 scripts/cms/cms.py plan      # + prioritized worklist (mechanical/substantive)
python3 scripts/cms/cms.py status    # terminal dashboard
python3 scripts/cms/cms.py all       # index + analyze + plan

# or via make
make cms-status   cms-index   cms-analyze   cms-plan   cms-all
```

### Review in Docker (no host Python needed)

```bash
docker compose -f docker-compose.cms.yml run --rm cms          # full review: tests + build + dashboard + mechanical preview
docker compose -f docker-compose.cms.yml run --rm cms status   # dashboard only
docker compose -f docker-compose.cms.yml run --rm cms test     # unit tests
docker compose -f docker-compose.cms.yml run --rm cms shell    # interactive shell
```

This is a self-contained `python:3.12-slim` environment (no `.env`, theme, or
Ruby). Outputs land under `.cms/` on your host via the bind mount.

## Lanes: mechanical vs substantive

Every detected issue is tagged with a **lane**:

- **Mechanical** — deterministic, fixed in one pass by
  `make content-normalize-apply` (string→list coercion, ISO dates, draft
  booleans, quest keyword shape, etc.). Safe to **auto-merge** after green CI.
- **Substantive** — needs authoring/judgment (missing description, thin content,
  SEO rewrites, broken-link repair, new content). Goes to a **reviewed PR**.

This split is what lets the daily loop fast-track safe fixes while keeping
content changes under human review. See `config.yml › loop`.

## Safety: what the loop must never touch

`config.yml` declares:

- `read_only` — vendored upstream content (e.g. OverTheWire wargame walkthroughs,
  detected by `source_repo`/`source_url` frontmatter + the `wargames/**` glob).
  Indexed for visibility, **never rewritten**.
- `read_only.structural_globs` — READMEs, quest templates/tools/codex. Indexed
  but exempt from deep field checks (matches the CI validator's skip list).
- `generated` — registry-derived data (`_data/quests/*.yml`, navigation,
  statistics). **Never hand-edited**; regenerated with `make quest-data`.

## Editing the policy

`config.yml` is the only file you hand-edit. Common knobs:

- `collections.*.recommended` / `required_extra` — per-collection field contract.
- `constraints` — SEO length limits and word counts (kept in sync with
  `.github/content-review-config.yml`).
- `loop.batch_size` / `mechanical_batch_size` — how much each daily run proposes.
- `loop.collection_priority` — which collections the loop prioritizes.
- `read_only` / `generated` globs — the safety boundaries above.
